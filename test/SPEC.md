# 回帰テストの仕様

このディレクトリは、[site/index.html](../site/index.html)(想定利用シーン/5つの価値セクション)が約束している5つの利用シーンが、実際に自然言語での問いかけとして機能するかを検証する回帰テスト一式。各シナリオの実装(問いかけ文・成功基準・判定ロジック)は`scenario-N.sh`自身に持たせてあり、このファイルは「なぜこのテストケースが存在するか」と「何を確認しているか」だけを一段上から説明する。プロンプトや判定基準の正確な文言はコード側が単一の真実源であり、ここでは重複させない。

実行方法・設計上の制約(合成フィクスチャを作らない理由、実行モデルと判定モデルを分ける理由等)は[lib.sh](lib.sh)のヘッダーコメントを参照。実行結果は`~/.local/share/claude-observability/regression-results.jsonl`に蓄積される。

## シナリオ1: 離席中・自律実行の振り返り

- **存在意義**: バックグラウンドタスクや自律実行は、それを離れると Claude Code 自身も文脈を覚えていない。別セッションから後で振り返れることが価値の根幹。
- **確認観点**:
  - ヒントなしで、別の実行単位(サブエージェント)の内容だと正しく特定できるか
  - 実行内容が実際のログからの要約として返るか(記憶ではなく)
  - 「想定外のことはなかったか」という副次的な問いにも答えられるか
- **実装**: [scenario-1.sh](scenario-1.sh)。唯一、明示的な準備(使い捨てサブエージェントの起動)を伴う——振り返る対象が実在することが前提のため。
- **既知の限界(2026-07-09確認)**: このシナリオを、生きた対話セッション(このリポジトリを操作している最中の会話そのもの)から`run-all.sh`経由で実行すると、判定が当てにならないことがある。理由は、その対話セッション自身のJSONLがコマンド実行のたびに書き込まれ続けるため、直前に起動した使い捨てサブエージェントより常に新しく見えてしまうこと。実行内容の特定先が変わるだけで daimon 自体の不具合ではないが、この条件下では「サブエージェントをヒントなしで見つけられるか」という確認観点1つは検証できない。無人・非対話的な実行(例: 生きた会話が存在しない状態での cron 実行)であればこの限界は生じないと考えられるが未検証。

## シナリオ2: 過去の対話の想起

- **存在意義**: 「前に相談した件、結局どうなった?」に、セッションIDや日付を指定せず答えられること。
- **確認観点**:
  - トピックの手がかりだけで該当セッションの内容と一致する応答が返るか
  - 実際に決定・結論があった場合、理由も含めて答えられるか
  - 生ログの引用ではなく要約になっているか
- **実装**: [scenario-2.sh](scenario-2.sh)。題材は固定せず`lib.sh`の`pick_recall_topic`が実行時に実データから選ぶ(固定題材が他の文書に書き写されて陳腐化した経緯は同ファイルのコメント参照)。

## シナリオ3: 手直し・訂正傾向の可視化

- **存在意義**: どの作業を安心して任せられるか、まだ細かい指示が要る領域はどこかを、印象論ではなく機械的なシグナルから答えられること。
- **確認観点**:
  - `AskUserQuestion`の応答実績のような具体的なシグナルに基づいているか
  - 何らかの内訳・分類を示せているか
  - 手法自体の限界(暗黙の訂正は拾えない等)を過度に断定的に扱っていないか
- **実装**: [scenario-3.sh](scenario-3.sh)。合成フィクスチャは作れない(`AskUserQuestion`は`claude -p`下で無条件拒否される、`lib.sh`参照)ため、既存の実履歴のみに依拠する。

## シナリオ4: コスト・トークン消費の実績把握

- **存在意義**: 「今日/今週/今月どれだけ使ったか」に、チャット上の数値とダッシュボードへのリンクの両方で答えられること。
- **確認観点**:
  - 具体的なコスト数値とモデル別内訳が含まれるか
  - Grafanaダッシュボードへのリンクが同じ応答に含まれるか
  - 「今日」が暦日として扱われているか(ローリング24時間と混同していないか)
- **実装**: [scenario-4.sh](scenario-4.sh)。

## シナリオ5: 積み重ねてきた実績の実感

- **存在意義**: これまでの利用実績(セッション数・累計時間)を、一人で進めてきた作業でも自分自身で確認・実感できること。
- **確認観点**:
  - セッション数が既知の不具合のある集計方法(`increase()`)に基づいていないか
  - Prometheusの保持期間を超える「全期間」の問いに、その制約を正直に説明しているか
  - ダッシュボードへのリンクが含まれるか
- **実装**: [scenario-5.sh](scenario-5.sh)。
