# PRFAQ: daimon — Claude Code のためのローカル自己観測ツール

## プレスリリース

### Claude Code というツールに何を任せ、どう使いこなしてきたかを、後から確認できるようにする

daimon は、Claude Code を日常的に使う開発者が、このツールに何を任せ、それがどう働いたか——対話の記録、離席中や自律実行の実態、費やした時間とコスト、手直しをした場面、積み上げてきた実績——を、Claude Code 自身に自然言語で問いかけるだけで確認できるようにするローカル完結のツールです。

Claude Code を使い込み、任せる範囲がバックグラウンドタスクやスケジュール実行にまで広がるほど、開発者はこの強力なツールを使いこなす一方で、その振る舞いを後から検証する責任も負うことになります。しかし今のところこの検証は簡単ではありません。

- 別のセッションやバックグラウンドで動いていた Claude Code は、結局どこまで実現してくれていたのか。そのセッションを離れてしまうと、もう覗く術がない。
- 前にこの件を Claude Code にどう相談し、どう決着したのか。
- Claude Code にどの作業を安心して任せられるようになったか、逆にまだ細かい指示が要る場面はどこか。
- 今月、Claude Code の利用にどれだけのコストがかかっているのか。
- これまで Claude Code をどれだけ使い込んできたのか。

このうち、離席中の振り返り・過去の対話の想起・手直し傾向の把握は、会話ログがどこにあり、どう読み解けばよいかを都度説明すれば、Claude Code 自身に自力で答えさせることもおそらくできますが、毎回同じ説明をし直す手間がかかり、答えもプレーンテキストの羅列に留まります。一方コスト・トークン消費や積み重ねてきた実績は性質が異なり、そもそもその記録自体がどこにも存在しないため、daimon が仕組みを用意して初めて答えられるようになります。

これらは表面上は別々の問いに見えますが、根は一つです。「Claude Code というツールに何を任せ、それがどう働いたかを、後から確認できること」——これさえできれば、離席中にどこまでやってくれていたかも、前にどう決着したかも、どれだけ手直しをしてきたかも、どれだけの時間とコストを費やしてきたかも、どれだけ使い込んできたかも、すべて同じ一つの行為(確認)が向く先の違いに過ぎません。

daimon は、この確認を Claude Code 自身への自然言語の問いかけだけでできるようにします。Claude Code 公式の OTel テレメトリを有効化してローカルの Prometheus/Loki/Grafana スタックへ流し込み、会話全文を扱う専用の経路もあわせて用意した上で、公式の Grafana MCP サーバーと専用 Skill を通じて、これらすべてに自然言語で問い合わせられるようにします。実装の詳細をその都度教える必要はなく、コストや実績のような値はダッシュボードへのリンクを介して見やすく確認できます。ただし、離席中の振り返り・過去の対話の想起・手直し傾向の把握については、その都度実装を教えて答えさせる方法と比べて daimon の回答がどれだけ正確・高速かは、今のところ検証されていません。

セットアップは `./setup.sh` 一発。完全にローカル・単一ユーザー前提で動作し、外部への送信や長期保管、他ユーザーとの共有は一切行いません。有効化・無効化はいつでも切り替えられる任意機能です。

「寝ている間に別セッションで走らせていたタスクを、翌朝あらためて別のセッションを開いてから『どこまでやってくれた?』と聞くのが楽しみになった」— 想定利用者(自律実行機能を使い始めた個人開発者)

「先週相談してた設定変更、結局どっちに決めたんだっけ、と聞いたら該当のセッションを探し出して要約してくれた」— 想定利用者(複数プロジェクトを掛け持ちする個人開発者)

「どの作業は安心して任せられるようになったか教えてもらって、次のプロジェクトでは最初から細かく指示すべき場面を先に見極められるようになった」— 想定利用者(自律実行機能を使い始めた個人開発者)

「毎月のコストがどの作業に由来しているか Claude Code に聞いたら、要約と一緒にダッシュボードへのリンクが返ってきて、内訳をグラフで一目で確認できた」— 想定利用者(コストを気にしながら使う個人開発者)

「一人で黙々と進めてきた作業だから誰も気づいてくれないけど、これまで何セッション、どれだけの時間 Claude Code を使い込んできたか自分で確認してみたら、地味に積み上げてきたんだなと実感できた」— 想定利用者(長期間同じプロジェクトを Claude Code と進めている個人開発者)

### 今すぐ始めるには

```bash
git clone https://github.com/mahitotsu/daimon.git
cd daimon
./setup.sh
```

Claude Code を再起動すれば、次回のセッションから自然言語での問い合わせが使えるようになります。

## FAQ

### 誰のためのツールですか?

Claude Code を日常的に、複数のプロジェクト・複数セッションにわたって使い込んでいる個人開発者向けです。同じプロジェクトを長く続けている人、バックグラウンドタスクや cron・リモートトリガーなど見ていない間にも Claude Code が動く機能を使っている人ほど、確認したい対象が増えるぶん価値が大きくなります。

ただし対象は、**自分が管理する端末**で使う場合に限ります。会社貸与 PC や共有環境、顧客の機密情報を扱うプロジェクトでは、後述のプライバシー上の理由から推奨していません。業務でヘビーに使い込んでいる人ほどこの条件に当てはまらないことも多いはずですが、これは意図した線引きです。暗号化・保存期間の上限・対象ディレクトリの除外といった緩和策を持たない設計を選んだ以上、個人が所有する環境での利用に絞っています。チーム/組織での利用統制、監査証跡、コンプライアンス対応も非スコープです(README.md 冒頭にも明記)。

### 具体的にどんな悩みに応えますか?

根は一つです。「Claude Code というツールに何を任せ、それがどう働いたかを、後から確認できること」。この一つの行為が向く先によって、以下の5つの価値として現れます。

1. **離席中・自律実行の振り返り** — バックグラウンドタスクや cron 実行は、それが動いていたセッションを離れると Claude Code 自身もう文脈を覚えていません。翌朝開いた新しいセッションからでも、「昨夜のタスクは結局どこまで実現してくれていたのか」を Claude Code に聞くだけで確認できます。想定外のことがなかったかも、同じ問いの中で自然に確認できます。
2. **過去の対話の想起** — 「以前 X について相談した時、どんな結論になって何をしたか」を、セッションをまたいで全文検索・要約させられます。Claude Code 自体の要約的な記憶機能を補う形で、加工されていない生の全文に立ち返れる点に価値があります。
3. **手直し・訂正傾向の可視化** — 「どの作業は安心して任せられるようになったか、逆にまだ細かい指示が要る場面はどこか」を確認できます。次のプロジェクトでどこを細かく指示すべきか、先に見極める判断材料になります。
4. **コスト・トークン消費の実績把握** — 「今日/今週/今月、どのモデルにどれだけ使ったか」「先週と比べて増えているか」を、チャット上の要約と、詳細を見るための Grafana ダッシュボードへのリンクで答えられます。数字を並べた文章より、モデル別・期間別の内訳はグラフで眺める方が素直に頭に入ります。
5. **積み重ねてきた実績の実感** — これまでどれだけ Claude Code を使い込んできたか(セッション数・トークン量・費やした時間の推移)を確認できます。チームで働いていれば朝会やレビューで自然に積み重ねに気づいてもらえますが、一人で Claude Code と向き合って進めている作業は、誰かが気づいて労ってくれるものではありません。その積み重ねを自分自身で確認し、実感するための価値です。数字を文章で読むより視覚的に眺める方が実感しやすいため、チャットでは一言の要約とダッシュボードへのリンクを返します。

離席中の振り返り・過去の対話の想起・手直し傾向の可視化は、いずれも数字の羅列ではなく物語や判断材料としての性質が強いため、チャット上の自然言語での回答が中心になります。コスト・トークン消費の把握と、積み重ねてきた実績の実感は、チャット上の要約と Grafana ダッシュボードへのリンクを組み合わせた体験になります。

### 何ができないのですか?(非スコープ)

- チーム/複数ユーザーでの利用統制、利用状況の横断比較
- クラウドへの転送・長期保管・改ざん検知
- Kubernetes 等でのオーケストレーション、強制的なテレメトリ有効化
- Claude Code 自体の応答品質・振る舞いの変更(daimon は観測するだけで、Claude Code の動作には一切介入しません)

### 導入に何が必要ですか?

Linux(WSL2 含む)で `systemd --user` が使え、かつ Docker が動く環境が必要です。詳細な前提条件は README.md の「対象環境」を参照してください。

### プライバシー・セキュリティ上の注意点は?

**ここが最も重要な注意点です。** Claude Code の公式テレメトリは、ユーザープロンプトやツールの引数・実行結果本体が外部に出ないよう、既定でガードされています。daimon はセッションの JSONL ファイル(会話全文・ツールの入出力を含む)を直接ローカルの Loki に再保存する経路を持っており、**この公式ガードを実質バイパスする設計**です。暗号化はされず、保存期間も無制限(手動削除のみ)です。

単一ユーザーが自分のローカル環境で使う前提であれば実害は限定的ですが、以下のケースでは導入を推奨しません:

- 会社貸与 PC や、自分以外のユーザーもアクセスしうる共有環境
- 顧客データや機密情報を扱うプロジェクトで、その内容がまるごとログに残ることが許容できない場合

有効化・無効化は `~/.claude/settings.json` の `env` でいつでも切り替えられます。詳細は README.md の「セキュリティ/プライバシー上の注意」を参照してください。

### 導入コストや運用の手間は?

`./setup.sh` を実行してから Claude Code を再起動するだけです。再実行しても安全(冪等)です。ただし、`systemd --user` はログインセッションが途切れると停止するため、バックグラウンドでも収集を継続したい場合は別途 `loginctl enable-linger` の設定が必要です(root 権限が要るため自動化していません)。アンインストール手順は README.md に記載しています。

### 会話全文はどこまで正確に検索できますか?

セッションの JSONL tail 経由の全文検索(`{job="claude-code-sessions"}`)は元データをそのまま保持しているため、トランケートなしで検索できます。一方、OTel 経由のログ(`{service_name="claude-code"}`)は集計用途向けで、内容はサイズ上限つきでトランケートされます。どちらを使うべきかは Skill が自動的に判断します。

### トークン/コストの数値はどのくらい信頼できますか?

Claude Code 公式の OTel メトリクスをそのまま使っているため、公式 CLI が把握している値と一致します。ただし、メトリクス名や属性は Claude Code のバージョンによって変わりうるため、応答時に「このバージョンで確認した」という前提を Skill 側で明示するようにしています。

### まだ検証できていないことは何ですか?

2026-07-08 に、このドキュメントが約束した5つの想定利用シナリオを実際に問いかけて検証しました(`docs/scenarios.md` 参照)。いずれも自然言語での問いかけが期待通りに機能することを確認済みです(検証の過程で、Alloy がサブエージェントの実行内容を収集し損なっていた設定バグなど、複数の実装上の問題も発見・修正しています)。

ただし以下は今もなお未検証です:

- 離席中の振り返りは、Agent/Task サブエージェントを使ったバックグラウンド実行では確認できましたが、cron・リモートトリガーのような完全に独立した別プロセスのセッションでの実例はまだ検証していません。
- 手直し・訂正傾向の可視化は、`AskUserQuestion` の応答結果を代理指標として使える設計を確立できましたが、質問を介さない暗黙の訂正(黙って書き直す等)までは捉えられておらず、サンプル数もまだ小さいため、傾向として安定して信頼できるかは今後の使用量の蓄積次第です。
- そして根本的に、離席中の振り返り・過去の対話の想起・手直し傾向の把握について、その都度実装を教えて Claude Code 自身にその場で答えさせる方法と比べて、daimon 経由の回答がどれだけ正確・高速かは依然として未検証です。今回確認したのは「daimon が想定した経路が実際に動くこと」であり、「その経路が代替手段より優れていること」ではありません。実際、今回の検証中に見つかった実装バグ(Alloy の非再帰グロブ)や記述の誤り(Prometheus リテンションの誤記)は、daimon の経路をそのまま信じたのでは見つからず、設定ファイルを直接読む・`curl`でAPIを直接叩く・生JSONLを直接パースするといった、まさに「その都度実装を確認する」側の作業によって発見しています。つまり daimon は「経路が正しく機能している間」は速く・自然な回答を返せますが、その正しさ自体を daimon が自己検証する手段は今のところ無く、`docs/scenarios.md` のような ad hoc 相当の点検を定期的に行う人間(またはエージェント)側の作業に依存しています。
