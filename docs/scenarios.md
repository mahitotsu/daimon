# 検証シナリオ

`docs/prfaq.md` が約束した5つの価値それぞれについて、「想定利用者が実際に Claude Code へ自然言語で問いかけたときに、約束通りの体験が返ってくるか」を検証するための手順書。

## この文書の位置づけ

- **test-first で書く。** ここに書く手順は「PRFAQ が約束した、あるべき体験」を先に定義するものであり、現時点の実装(Skill・ダッシュボード・データ経路)がそれを満たす保証はない。書いた時点、あるいは初回実行時点で失敗して構わない。
- **失敗したら手順ではなく実装を疑う。** 各シナリオの成功基準は PRFAQ の文言から直接導いたものなので、基準そのものを緩めて実装に合わせるのではなく、実装(Skill の記述、クエリ、ダッシュボード)を基準に合わせて直していく。ただし、基準が PRFAQ の記述と食い違っていたり曖昧だったりすることが判明した場合は、その旨を明記した上で基準側を直すこともありうる。
- 各シナリオは「想定する問いかけ」「期待される経路」「成功基準」「現状」の4項目で構成する。「現状」は検証を実施するたびに更新し、確認日・Claude Code バージョン・結果(成功/部分成功/失敗とその理由)を記録する。
- 検証は実際に Claude Code のセッションから該当の問いかけを行い、返ってきた応答を人間が読んで成功基準と照合する形で行う(自動テストではない)。

## 前提

各シナリオを検証する前に、README.md の「検証手順」が一通り通っていること(otel-lgtm 起動、Loki/Prometheus 疎通、Alloy 稼働、実データ到達)を確認しておく。ここが崩れている状態でシナリオを検証しても、Skill 側の問題なのか基盤側の問題なのか切り分けられない。

## シナリオ1: 離席中・自律実行の振り返り

PRFAQ より:
> 「寝ている間に別セッションで走らせていたタスクを、翌朝あらためて別のセッションを開いてから『どこまでやってくれた?』と聞くのが楽しみになった」

### 想定する問いかけ

バックグラウンドタスクや cron・リモートトリガーで Claude Code を走らせておき、それとは別の新しいセッションから:

- 「昨夜(または直前に)走らせていたタスクは結局どこまでやってくれた?」
- 「さっきのバックグラウンドセッション、何か想定外のことは起きてない?」

### 期待される経路

`claude-observability` Skill が「セッション内容検索」の手順(`{job="claude-code-sessions"}` への LogQL、対象セッションの特定は mtime やタイムスタンプ境界で絞り込み)を使って、元セッションの JSONL 全文から該当タスクの実行内容を読み取り、要約して返す。

### 成功基準

- [ ] 現在のセッションとは別のセッション ID の内容であることを、Claude Code 自身が(ユーザーに指定させずに)正しく特定できる。
- [ ] 実行された作業内容(何をしたか)が、Claude Code 自身の記憶ではなく JSONL 全文からの要約として返る(=セッションをまたいでいるにもかかわらず答えられる)。
- [ ] 「想定外のことがなかったか」という副次的な問いにも、同じ会話の中で(別途データソースの説明をユーザーがし直すことなく)答えられる。
- [ ] 応答がプレーンな引用の羅列ではなく、要約された自然言語になっている。

## シナリオ2: 過去の対話の想起

PRFAQ より:
> 「先週相談してた設定変更、結局どっちに決めたんだっけ、と聞いたら該当のセッションを探し出して要約してくれた」

### 想定する問いかけ

- 「先週 X について相談した時、結局どう決着した?」
- 「前に Y の設定を変えるかどうか議論したと思うけど、何が理由でどう決めたんだっけ?」

セッション ID やファイル名など、内部的な手がかりは一切与えない。

### 期待される経路

`{job="claude-code-sessions"}` への全文検索(負の line filter で `tool_result`/`toolUseResult` を除外しつつキーワードや時期で絞り込み)によって該当セッションを特定し、該当箇所を読み取って要約する。

### 成功基準

- [x] ユーザーが「いつ・どのセッション」かを指定しなくても、キーワードや相対的な時期の手がかりだけから該当セッションを見つけられる。
- [x] 議論の結論(何をどう決めたか)だけでなく、決めた理由も拾えている。
- [ ] 複数セッションにまたがって同じ話題が出てきた場合、最新の結論(または該当する方)を取り違えない。(今回はヒットが単一セッションに閉じていたため未検証)
- [x] 生ログの引用そのままではなく、要約として返る。

### 現状

確認日: 2026-07-08、Claude Code v2.1.162。
結果: 成功(検証を通じて SKILL.md の記述ギャップ1件を発見・修正)。

- 実際の題材: 「ツール呼び出し頻度・所要時間の集計方法をどうするか」という、この数日で本当に交わされた相談(2026-07-07 のセッション)を対象に、セッションIDや日付を与えずキーワード(`TraceQL`)だけで検索。
- 該当セッションを特定し、結論と理由の両方を要約できた: 「チャット側の頻度・所要時間・ターン構造の分析は Loki 手組みから Tempo TraceQL へ移行したが、Grafana ダッシュボードのパネルは意図的に Loki のまま残した」。理由も2点拾えた——(1) TraceQL metrics クエリには3時間の範囲上限があり、ダッシュボードの自由な時間ピッカーと相性が悪い、(2) 新しく増えた価値(waterfall・遅いターンの原因調査)は本質的に1トレース単位の深掘りで、既存の「見出し数値はダッシュボード、深掘りはチャット」という設計方針にすでに沿っている。「Tempoに全面移行した」のような単純化された誤った要約にはならなかった。
- 副産物として SKILL.md の抜けを発見: `{job="claude-code-sessions"}` はプロジェクトを問わず `~/.claude/projects/*` 全体をタイルしているため、プロジェクトを絞らずに検索すると無関係な別プロジェクト(`charon`)のセッションが混入した。**「セッション内容検索」節に、既定でカレントプロジェクトの `filename` にスコープする指針を追記した**(`.claude/skills/claude-observability/SKILL.md`)。
- 未検証項目: 「同じ話題が複数セッションにまたがる場合の取り違え」は、今回のヒットが単一セッションに閉じていたため確認できていない。該当する事例が見つかった際に再検証する。

## シナリオ3: 手直し・訂正傾向の可視化

PRFAQ より:
> 「どの作業は安心して任せられるようになったか教えてもらって、次のプロジェクトでは最初から細かく指示すべき場面を先に見極められるようになった」

### 想定する問いかけ

- 「これまでの作業で、Claude Code に安心して任せられている領域と、まだ細かい指摘・訂正が多い領域はどこ?」
- 「この1ヶ月でどんな作業に訂正・やり直しが多かった?」

### 期待される経路(2026-07-08 設計・実装)

当初、「訂正傾向」を判定する明確なデータソースは README.md / SKILL.md のどこにも定義されておらず、設計自体が固まっていなかった。検証を通じて以下の経路を設計・SKILL.md に追記した:

1. まず `tool_decision=reject` を候補として実データを確認したが、15日間で678件中2件のみと機能しないことが判明(このプロジェクトの運用は `acceptEdits` が中心で、個々の編集が承認プロンプトを経ないため)。
2. 代わりに **`AskUserQuestion` の応答結果**を使う。JSONL の `toolUseResult` フィールドに、提示した選択肢(`options[].label`、うち1つに `(推奨)`/`(Recommended)` マーカーが付く)と実際の回答(`answers`)が同じ行に同居しており、追加の計装なしに以下の3分類ができる:
   - **recommended_accepted**: 回答が「(推奨)」付きの選択肢ラベルと完全一致
   - **alternative_selected**: 回答が他の用意された選択肢ラベルと一致
   - **other_custom**: どの選択肢ラベルとも一致しない自由記述——Claude が用意した見立てそのものをユーザーが受け入れなかった、最も強いシグナル
3. クエリは `{job="claude-code-sessions", filename=~".*-<project>/.*"} |~ "toolUseResult" |~ "\"answers\""` で候補行を絞り込み、行ごとに JSON をパースして分類する。

### 成功基準

- [x] 「訂正・やり直し」に相当する具体的なシグナルが、Loki から機械的に定義・抽出できる(`AskUserQuestion` の `toolUseResult` 解析)。
- [ ] その結果が「作業の種類・領域」ごとに分類されて返ってくる(今回のサンプル数=12では領域別に割るには小さすぎ、時期・トピック別の分類は今後件数が増えてから)。
- [x] 応答が「安心して任せられる/まだ指示が要る」という PRFAQ の言葉に対応する形の判断材料になっている(recommended_accepted の比率が高いほど「安心して任せられる」、other_custom が多いほど「まだ指示が要る」の代理指標として機能)。

### 現状

確認日: 2026-07-08、Claude Code v2.1.162。
結果: 成功(設計を新規に確立し、実データで検証)。

このリポジトリ自身の過去15日間・12件の `AskUserQuestion` を全件確認:

| 分類 | 件数 |
|---|---|
| recommended_accepted | 6 |
| alternative_selected | 2 |
| multiSelect 部分一致 | 1 |
| **other_custom** | **4(33%)** |

other_custom 4件のうち2件は、単に別の答えを選んだだけでなく **Claude 自身の推奨判断そのものへの異議**だった(例:「結局、やった方がいいのか、やめたほうがいいのか、どっちだとあなたは言っていますか」「〜というメリットがあるのではなかったのですか?」)。さらに、まさに本セッション中でシナリオ3の進め方を尋ねた際にも、ユーザーが用意した3択のどれでもなく Other を選んでこの `AskUserQuestion` 分析というアイデア自体を提案しており、これが4件目の other_custom として観測に含まれている——検証対象の手法が、検証の実施中にリアルタイムで実例を生んだ形になった。

この設計・実装を `.claude/skills/claude-observability/SKILL.md` に「Correction-tendency signal: AskUserQuestion outcomes」節として追記した。残る制約: この手法は Claude が `AskUserQuestion` を使った場面しか捉えられず、質問を介さない暗黙の訂正(黙って書き直す、後から元に戻す等)は対象外——「訂正傾向」の下限値であり全量ではない。サンプル数(n=12)も現時点では作業領域別の内訳を出すには小さすぎるため、件数が増えた時点で再集計する。

## シナリオ4: コスト・トークン消費の実績把握

PRFAQ より:
> 「毎月のコストがどの作業に由来しているか Claude Code に聞いたら、要約と一緒にダッシュボードへのリンクが返ってきて、内訳をグラフで一目で確認できた」

### 想定する問いかけ

- 「今月のコストはどれくらい?」
- 「先週と比べて今週は使用量が増えてる?」
- 「今日のコストはどのモデルの利用が多い?」

### 期待される経路

SKILL.md の「Usage summary dashboard」節の手順どおり、`query_prometheus` で見出しの数値(合計コスト等)を1〜2回取得し、`generate_deeplink` で `claude-code-usage-summary` ダッシュボードへのリンクを該当期間で生成して返す。

### 成功基準

- [x] チャット上の要約(具体的な数値)と、Grafana ダッシュボードへのリンクの両方が同じ応答に含まれる。
- [x] リンクの時間範囲が、問いかけた期間(今日/今週/今月)と一致している。
- [x] 比較の問い(「増えているか」)にも、2つの期間のクエリを行って比較した結果を数値で返せる(※週次粒度は現時点でデータ不足のため未検証、日/24h粒度では確認済み)。
- [x] 見出し数値がダッシュボード上の該当パネルの値と一致する(手計算のズレがない)。

### 現状

確認日: 2026-07-08、Claude Code v2.1.162。
結果: 成功(週次比較のみ、データ蓄積待ちで再検証が必要)。

- 「今日のコストは?」に相当する `sum(increase(claude_code_cost_usage_USD_total[24h]))` で合計 $9.85、モデル別内訳(`claude-sonnet-5`: $9.69、`claude-haiku-4-5-20251001`: $0.16)を取得し、同じ応答で `generate_deeplink`(`claude-code-usage-summary`、`now-24h`〜`now`)によるダッシュボードリンクも生成できた。要約+リンクの組み合わせは成立(基準1・2)。
- `get_dashboard_panel_queries` で「コスト(選択期間の合計、USD)」パネルの PromQL が手動クエリと同一の式(`sum(increase(claude_code_cost_usage_USD_total[$__range]))`)であることを確認済み。同じ時間範囲なら値が一致する設計になっている(基準4、式レベルでの確認。実際のパネル描画は未確認)。
- 「先週と比べて増えているか」の比較(基準3)は、7日粒度では未達だったが、**日/24時間粒度に落とせば達成できることを確認した**(2026-07-08 追加検証)。直近7日分(`now-7d`〜`now`)は取得できたが、その1週間前(`now-14d`〜`now-7d`)は空振り(`no data`)——これは実装のバグではなく、このオブザーバビリティ基盤自体を数日前に構築したばかりで14日分のデータが無いため。一方、直近24時間(`now-24h`〜`now`: $10.74)とその前の24時間(`now-24h`を起点に`now-24h`〜`now-48h`: $35.64)、および昨日1日分(暦日、`endTime: now/d` で `[24h]`: $19.25)はいずれも値が取れ、比較が成立した。7日分の履歴という高いハードルではなく、直近1〜2日という現実的な粒度でまず「比較できるか」を確認するのが妥当。14日分のデータが蓄積された後に週次比較も再検証する。
- 気づいた点 → 対応済み: 「今日」を `now-24h`〜`now` のローリング窓として扱っており、暦日(0時起点)ではなかった。PRFAQ の「今日のコスト」という自然文の期待と食い違うため、2026-07-08 に「今日 = ローカル深夜0時〜現在」のカレンダー日へ定義を変更した(`.claude/skills/claude-observability/SKILL.md` の「Usage summary dashboard」「Token / cost usage」節を修正)。`generate_deeplink` は Grafana ネイティブの `now/d` をそのまま渡せるが、`query_prometheus` の instant query は PromQL の duration リテラルが必要なため、ローカル深夜0時からの経過時間をエージェント側で計算して埋め込む方式にした。
  - 再検証(2026-07-08 00:15 JST、Claude Code v2.1.162): `increase(claude_code_cost_usage_USD_total[15m])`(深夜0時から15分経過)で $1.15、`generate_deeplink` に `{"from": "now/d", "to": "now"}` を渡してダッシュボードリンクも生成できた。同時刻に計測したローリング24時間の値($9.85、シナリオ4検証の1回目)とは明確に異なる数値になっており、定義変更が実際に意味のある違いを生むことを確認できた。
  - 「今週」「今月」は今回はスコープ外とし、ローリング窓(`now-7d`/`now-30d`)のまま据え置いた。同様のズレが指摘されたら改めて検討する。

## シナリオ5: 積み重ねてきた実績の実感

PRFAQ より:
> 「一人で黙々と進めてきた作業だから誰も気づいてくれないけど、これまで何セッション、どれだけの時間 Claude Code を使い込んできたか自分で確認してみたら、地味に積み上げてきたんだなと実感できた」

### 想定する問いかけ

- 「これまで Claude Code をどれだけ使い込んできた?セッション数と使用時間を教えて」
- 「今までの累計利用実績をダッシュボードで見たい」

### 期待される経路

`claude_code_session_count_total` / `claude_code_active_time_seconds_total` を SKILL.md の注意点(session_count は `increase()` ではなく distinct series カウント、かつ下限値として扱う)に従って集計し、チャット上の一言要約 + ダッシュボードへのリンクを返す。

### 成功基準

- [x] セッション数が SKILL.md に明記された正しい集計方法(`count(count by (session_id) (...))`)で算出されており、`increase()` の既知の不具合を踏んでいない。
- [x] セッション数がおそらく過小評価であることの注記が必要に応じて言及される、または少なくとも明らかに実態と乖離した数値(例: 0)を鵜呑みにして返さない(検証を機に「常にクロスチェック」へ SKILL.md を強化)。
- [x] 使用時間の累計が、保存期間(Prometheus のリテンション)の制約を超える「これまで全部」を問われた場合に、その制約をユーザーに説明できる(検証を機に SKILL.md にリテンション上限の記載を追加)。
- [x] ダッシュボードへのリンクが正しい時間範囲(全期間、またはリテンション上限)で生成される。

### 現状

確認日: 2026-07-08、Claude Code v2.1.162。
結果: 成功(ただし検証を通じて SKILL.md の記述ギャップ2件が見つかり、修正した)。

- セッション数: `count(count by (session_id) (last_over_time(claude_code_session_count_total[15d])))` で 21。SKILL.md 指定の正しい集計方法(`increase()` を使わない distinct series カウント)を使用(基準1)。
- クロスチェック用の `count(count by (session_id) (last_over_time(claude_code_token_usage_tokens_total[15d])))` は 23。21 と 23 という小さな乖離が実際に発生することを確認(基準2)。これは「明らかに低ければクロスチェックする」という従来の SKILL.md の書き方では見逃されうる程度の差だったため、**「常にクロスチェックし、token_usage 側の値を報告する」よう SKILL.md を修正した**(`.claude/skills/claude-observability/SKILL.md` の session-count 節)。
- 累計アクティブ時間: `sum(increase(claude_code_active_time_seconds_total[15d]))` で約9,659秒(≈2時間41分)。
- 「これまで全部」という問いへの対応(基準3): Prometheus の実際の retention を `curl localhost:9090/api/v1/status/flags` で確認したところ `storage.tsdb.retention.time: 15d`。**このリテンション上限について SKILL.md に一切記載がなかった**ため追記した。さらに、SKILL.md の既存記述は「7日のリテンション」と誤って書かれていた箇所があり(2026-07-06 時点の記述、実際は15d)、これも修正した。今回はこのツール自体が2026-07-06に構築されたばかり(2日前)で15日以内に収まるため、現在の「全期間」データはたまたま真の全履歴と一致しているが、この前提は15日を過ぎると崩れる。
- ダッシュボードリンク(`{"from": "now-15d", "to": "now"}`)は正しく生成できた(基準4)。

総括: 検証手順自体は正しく回るが、「累計・全期間」を扱うこのシナリオは、実装のクエリより **SKILL.md のドキュメント側の抜け・誤り(リテンション未記載、7d/15dの誤記、クロスチェック条件の緩さ)** の方が実害が大きかった。この2点を修正済み。

## 検証後の運用

各シナリオを実際に検証したら、「現状」節を次の形式で更新する:
